# ir-lang v0.2.0 — Core

**The intermediate representation, the lowering interface, and validation.** v0.1.0
stood up the crate and its quality gates; v0.2.0 is the first release with domain
logic. It defines the IR a compiler optimizes and lowers code through, the builder a
front-end lowers into it with, and the validator that proves a function is
well-formed. This is the hard part of the roadmap, and it is not deferred.

## What's in 0.2.0

### The IR

A `Function` is a control-flow graph of basic blocks in SSA form. Each block is a
straight-line run of value-producing instructions ended by exactly one terminator,
and every value is named by a small `Value` handle and defined exactly once. Blocks
and values live in flat arenas addressed by dense indices, so building and walking
the IR is a linear pass over contiguous memory and a pass can key a side table
directly on a handle. Values cross control-flow joins as block parameters — a `jump`
or `branch` passes one argument per target parameter — which keeps the representation
flat and removes the bookkeeping of phi nodes.

The surface is deliberately small: `Type` (`int`, `float`, `bool`, `unit`), the
`Value` and `Block` handles, `BinOp` and `UnOp`, the `Inst` and `Terminator` enums,
and the `Function` that holds them with read accessors and a textual `Display`.

### Lowering through the builder

There is no AST type here to lower *from* — a language brings its own syntax tree.
Lowering is expressed through the `Builder`: a front-end walks its tree and calls a
method per construct — a literal becomes a constant, an operator becomes an
instruction, an `if` becomes two blocks and a branch, a join becomes a block
parameter — and the builder hands back the `Value` each result defines. This is the
same shape as Cranelift's `FunctionBuilder` or LLVM's `IRBuilder`. Result types are
inferred from the operation, so they are never passed in.

### Validation

`Function::validate` confirms a function is well-formed in one pass, returning a
defined `ValidationError` rather than panicking. It checks the structural rules — one
terminator per block, branch targets that exist, argument counts and types that match
the target parameters, a numeric or boolean operand where the operation requires one,
the entry block never a branch target — and then the SSA dominance property: every
use of a value is reached by its single definition. Dominators are computed with the
Cooper–Harvey–Kennedy algorithm over the reachable graph, and both the dominator and
reachability walks use an explicit stack, so a deeply nested function cannot overflow
the call stack.

### Dependencies — none wired, by design

The crate is self-contained. An IR's machine-level type system (`int`, `float`,
`bool`, `unit`) is a different layer from a source language's types, the way LLVM
IR's types are independent of any front-end, so ir-lang defines its own `Type` rather
than depending on `type-lang`. And `ast-lang` is a generic syntax-tree substrate with
no concrete grammar to lower *from* — lowering is the builder, driven by the
consumer. Both dependencies are left unwired, with the reasoning recorded in
[`dev/ROADMAP.md`](https://github.com/jamesgober/ir-lang/blob/main/dev/ROADMAP.md)
under the anti-deferral rule. Mapping a front-end's AST and source types onto the IR
is the consumer's lowering step.

## Breaking changes

**None.** v0.1.0 had no public surface. Everything in this release is new. The
surface remains pre-1.0 and may change in a later 0.x minor before the `1.0` freeze.

## Verification

Run on Windows x86_64 and Linux (WSL2 Ubuntu), Rust stable and the 1.85 MSRV;
identical commands run in the configured CI matrix:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --all-features -- -D warnings
cargo test
cargo test --all-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo build --no-default-features
cargo +1.85 build --all-features
cargo deny check
cargo audit
```

All green. Counts at this tag: 30 unit tests, 5 lowering integration tests, 4
property tests (`proptest`), 2 serde round-trip tests (all features), and 45
doctests.
