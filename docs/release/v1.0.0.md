# ir-lang v1.0.0 — stable

**1.0.** The public API is frozen and stable. v0.1.0 stood up the scaffold and the
quality gates; v0.2.0 built the core — the SSA intermediate representation, the
`Builder` lowering interface, the textual `Display`, and `Function::validate`. This
release ratifies that surface as the `1.0` contract: it follows Semantic Versioning,
carries no breaking changes before `2.0`, and ships with the full property-test and
benchmark suite verified on Linux, macOS, and Windows.

There is no breaking change from `0.2.0` — a `0.2.0` program compiles and behaves
identically. The freeze adds only what hardens the release: a faster validator, a
complete check for hand-assembled and deserialized IR (one additive, non-breaking
error variant), runnable examples, and the stability documentation.

## What is ir-lang?

The intermediate representation a compiler optimizes and lowers code through. A
`Function` is a control-flow graph of basic blocks in SSA form: each block is a
straight-line run of value-producing instructions ended by one terminator, and every
value is named by a small handle and defined exactly once. There is no AST type to
lower *from* — a language brings its own tree and drives the `Builder`, the same shape
as Cranelift's `FunctionBuilder` or LLVM's `IRBuilder`. The crate is self-contained:
it defines its own machine-level `Type` and wires no first-party dependency.

```rust
use ir_lang::{Builder, BinOp, Type};

// fn max(a: int, b: int) -> int { if a < b { b } else { a } }
let mut b = Builder::new("max", &[Type::Int, Type::Int], Type::Int);
let a = b.block_params(b.entry())[0];
let bb = b.block_params(b.entry())[1];

let join = b.create_block(&[Type::Int]);
let then_blk = b.create_block(&[]);
let else_blk = b.create_block(&[]);

let cond = b.bin(BinOp::Lt, a, bb);
b.branch(cond, then_blk, &[], else_blk, &[]);
b.switch_to(then_blk);
b.jump(join, &[bb]);
b.switch_to(else_blk);
b.jump(join, &[a]);
b.switch_to(join);
let result = b.block_params(join)[0];
b.ret(Some(result));

let func = b.finish();
assert!(func.validate().is_ok());
```

## The 1.0 surface

The frozen contract is the IR core:

- **Types** — the machine-level `Type` (`Int`/`Float`/`Bool`/`Unit`), the `Value` and
  `Block` handles, and the `BinOp` and `UnOp` operation enums.
- **IR** — the value-producing `Inst`, the `Terminator` that ends a block, and the
  `Function` that holds them, with read accessors over its blocks, values, and
  control flow, and a textual `Display`.
- **Building** — `Builder`: `new`, `entry`, `current_block`, `create_block`,
  `block_params`, `switch_to`, the `iconst` / `fconst` / `bconst` constants, `bin`,
  `un`, the `ret` / `jump` / `branch` terminators, and `finish`. Result types are
  inferred from the operation.
- **Validation** — `Function::validate` and the `#[non_exhaustive]` `ValidationError`.
- **Serialization** — `serde` for every IR type under the `serde` feature.

The `Inst`, `Terminator`, `Type`, `BinOp`, `UnOp`, and `ValidationError` enums are
`#[non_exhaustive]` where they may grow, so additions stay non-breaking.

## SemVer promise

The crate follows [Semantic Versioning](https://semver.org):

- No documented item is removed or changed in a breaking way within `1.x`; breaking
  changes wait for `2.0`.
- New functionality is additive and arrives in minor releases. A new `ValidationError`
  variant, or a new instruction or terminator form, is a minor change.
- The MSRV is Rust `1.85`; raising it is a minor change, never a patch.
- Behaviour is part of the contract: a function that validates today keeps
  validating, and the textual `Display` form is stable for a given function.

The full statement lives in [`docs/API.md`](https://github.com/jamesgober/ir-lang/blob/main/docs/API.md#semver-promise).

## Breaking changes

**None.** `1.0.0` makes no breaking change to the `0.2.0` surface. The only addition
is the new `ValidationError::InconsistentDefinition` variant, which is non-breaking
because `ValidationError` is `#[non_exhaustive]`.

## Hardening since 0.2.0

- **Faster validation.** The SSA dominance check was reworked from a per-block
  allocation into a single linear pre-order walk of the dominator tree carrying one
  reused availability set. It now allocates nothing per block and scales linearly with
  the size of the function — validating a 3,000-block function in the same order of
  time it takes to build it.
- **A complete gate.** `validate` now checks the value table itself — handle ranges,
  definition sites, and recorded result types — so it is a sound check for a function
  assembled by hand or deserialized through `serde`, not only one the builder
  produced. A new `InconsistentDefinition` variant reports a value listed in a block
  that disagrees with its recorded definition.
- **No panics on malformed input.** A function with an out-of-range entry block, or no
  blocks at all, now returns a defined `BlockOutOfRange` error instead of panicking.

The dominator computation and the SSA dominance check were verified by an independent
adversarial review that fuzzed them against a brute-force reference over several
million random control-flow graphs and programs — including irreducible loops,
unreachable code, self-loops, and critical edges — with no soundness miss, no false
positive, and no panic.

## Verification

Run on Windows x86_64 and Linux x86_64 (WSL2 Ubuntu), Rust stable and 1.85 MSRV; the
same commands run across the CI matrix on Linux, macOS, and Windows:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --all-features -- -D warnings
cargo test
cargo test --all-features
cargo build --no-default-features
cargo +1.85 build --all-features
cargo build --examples
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo deny check
cargo audit
```

All green. Counts at this tag: 37 unit tests, 5 lowering integration tests, 4
property tests (`proptest`), 2 serde round-trip tests (all features), and 45
doctests, plus two runnable examples (`cargo run --example lower_ast`,
`cargo run --example validation`).

## What's next

The `1.0` surface is stable until `2.0`. Future work is documentation, tests, and
internal optimisation that does not change the public contract — plus the additive
candidates the design leaves room for: function calls (a `Call` instruction and an
opaque callee handle) and a multi-function module container, each a minor release
when it lands.
